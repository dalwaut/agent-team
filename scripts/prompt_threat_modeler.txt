## Enhanced Knowledge: STRIDE Analysis Patterns [security-scanning/stride-analysis-patterns]
- STRIDE categories: Spoofing (AuthN), Tampering (integrity), Repudiation (logging), Info Disclosure (confidentiality), DoS (availability), Elevation (AuthZ)
- Per-element threat mapping: External entities -> S,R; Processes -> all six; Data stores -> T,R,I,D; Data flows -> T,I,D
- Always identify trust boundaries first (Internet->DMZ, DMZ->Internal, App->DB), then analyze each crossing
- Risk score = Impact x Likelihood (1-4 scale each); Critical >= 12, High >= 6, Medium >= 3
- Key mitigations by category: S -> MFA + secure tokens; T -> input validation + parameterized queries; R -> tamper-evident audit logs; I -> encryption at rest/transit; D -> rate limiting + auto-scaling; E -> server-side RBAC + least privilege
- Find unencrypted data flows crossing trust boundaries -- high-priority gaps
- STRIDE per Interaction: apply to each (source_type, target_type) pair -- external->process gets all 6; process->datastore gets T,R,I,D
- Use DFD (Data Flow Diagram) to systematically enumerate all elements and flows before applying STRIDE

## Enhanced Knowledge: Attack Tree Construction [security-scanning/attack-tree-construction]
- Root goal = attacker objective; decompose into sub-goals via OR (any child) and AND (all children required) nodes; leaves are atomic attack steps
- Attribute each leaf: cost ($-$$$), time (hours/days/weeks), skill (low/medium/high), detection risk (low/medium/high)
- OR node: attacker picks easiest path (min difficulty); AND node: attacker must complete all steps (max difficulty, sum cost)
- Find easiest/cheapest/stealthiest paths to prioritize defenses -- block the lowest-cost attack path first
- Critical nodes = nodes appearing in the most attack paths; mitigating these gives highest coverage impact
- Coverage analysis: given mitigated attacks, calculate % of paths blocked vs. still open; report top 5 open paths
- Unmitigated leaf nodes (no mitigations listed) are the highest-priority defense gaps
- Include insider threats (requires_insider flag) and physical access attacks -- not all attackers are external
- Export to Mermaid/PlantUML for stakeholder communication; color-code leaves by difficulty (red=easy, blue=hard)
- Update attack trees regularly as threat landscape evolves; validate with red team exercises

## Enhanced Knowledge: Threat Mitigation Mapping [security-scanning/threat-mitigation-mapping]
- Control types: Preventive (stop attack before it occurs), Detective (identify attack in progress), Corrective (respond and recover)
- Control layers: Network (firewall, WAF, DDoS), Application (auth, validation), Data (encryption, ACL), Endpoint (EDR, patching), Process (training, IR)
- Defense in depth: every threat needs controls at 2+ layers AND 2+ control types (preventive + detective minimum)
- Coverage score = effectiveness x implementation_status; flag any threat with < 50% coverage as gap
- Unmapped threats = coverage 0% -- these are highest priority; single controls = single points of failure -- always flag
- Standard controls: MFA (AUTH-001), Input Validation (VAL-001), WAF (VAL-002), Encryption at rest (ENC-001), TLS (ENC-002), Security logging (LOG-001), RBAC (ACC-001), Rate limiting (AVL-001)
- Generate remediation roadmap: Phase 1 = critical threats with low coverage; Phase 2 = high-impact threats
- Test control effectiveness regularly with automated tests; controls degrade over time
- Track DORA metrics: deployment frequency, lead time, change failure rate, MTTR -- tie to control effectiveness
