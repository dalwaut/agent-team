AUDIT SCOPE:

1. **Sync I/O in Async Handlers (Critical — blocks event loop)**

   Scan all `async def` route handlers and `async def` functions in Python files:
   - Flag `open()`, `json.load()`, `json.dump()` called directly (not via asyncio.to_thread)
   - Flag `requests.get()`, `requests.post()` etc. — use `httpx.AsyncClient` instead
   - Flag `subprocess.run()`, `subprocess.Popen()`, `os.system()` — use `asyncio.create_subprocess_exec`
   - Flag `time.sleep()` — use `await asyncio.sleep()` instead
   - Flag `psycopg2` usage in async context — use `asyncpg` or `databases` library
   - Flag any synchronous DB client call inside an `async def`

2. **Unbounded Concurrency (Node.js)**

   Scan JavaScript files in tools/:
   - Flag `Promise.all([...])` over large arrays with no concurrency limit — use `p-limit` or batching
   - Flag `setInterval()` that launches async work without checking if previous run finished — can stack up
   - Flag event listeners (`.on('message', ...)`) that do blocking I/O synchronously
   - Flag `fs.readFileSync`, `fs.writeFileSync` in request handlers or event callbacks

3. **Missing Pagination**

   Scan route handlers and Supabase client calls:
   - Flag `.select()` on Supabase client with no `.limit()` on tables likely to grow (messages, conversations, tasks, audit_log)
   - Flag list endpoints returning unbounded arrays (no `limit`/`offset` or cursor params)
   - Flag `registry.json` and `queue.json` reads that load entire file into memory — flag if file could grow large

4. **Subprocess Spawning Without Semaphore**

   Scan orchestrator and squad runner code:
   - Flag `claude -p` subprocess spawns with no concurrency limit (check MAX_PARALLEL enforcement)
   - Flag `asyncio.gather()` over agent lists with no semaphore — can exhaust file descriptors
   - Flag subprocess spawning inside loops without throttling

5. **Memory Leak Patterns**

   Scan Python and Node.js services:
   - Flag caches (dicts, Maps) that grow without eviction or TTL
   - Flag global state that accumulates per-request (e.g., appending to a module-level list)
   - Flag WebSocket connections or event listeners that are never cleaned up on disconnect
   - Flag `asyncio.create_task()` calls where the task reference is not stored (task can be GC'd mid-run)

6. **Hot Path Analysis**

   Scan route handlers called on every request:
   - Flag JSON file reads (`team.json`, `config/orchestrator.json`) that happen on every request without caching
   - Flag repeated `Path.resolve()` or file existence checks in tight loops
   - Flag string formatting with `%` or `.format()` in log-heavy code paths (prefer f-strings or lazy logging)
   - Flag redundant DB queries where data could be cached for the request lifetime

7. **Service Configuration**

   Check service startup and config:
   - Flag missing `workers` count in uvicorn/gunicorn config — single worker = no parallelism
   - Flag missing `keepalive` timeout on uvicorn (default 5s — too short for long-running Claude CLI calls)
   - Flag Node.js services without `--max-old-space-size` when processing large datasets

## Enhanced Knowledge: Python Performance Optimization [python-development/python-performance-optimization]
- Profile before optimizing: use cProfile (CPU), line_profiler (line-by-line), memory_profiler (memory), py-spy (production sampling)
- List comprehensions are faster than append loops; generators use constant memory for large datasets
- String concatenation: use "".join(parts) not += in loops
- Dict/set lookups are O(1) vs list search O(n) -- use dict for membership checks on large collections
- Local variable access is faster than global -- copy globals to locals in hot loops
- functools.lru_cache for expensive recursive/repeated computations; check cache_info() for hit rates
- Use __slots__ on classes with many instances to reduce memory (no __dict__ overhead)
- multiprocessing.Pool for CPU-bound tasks; asyncio/aiohttp for I/O-bound tasks
- Batch DB operations: executemany + single commit vs individual insert+commit per row
- Use NumPy for vectorized numerical operations -- orders of magnitude faster than pure Python loops
- Detect memory leaks with tracemalloc: snapshot before/after, compare top allocations
- Use weakref.WeakValueDictionary for caches that should allow GC of unused entries

## Enhanced Knowledge: SQL Optimization Patterns [developer-essentials/sql-optimization-patterns]
- Always use EXPLAIN ANALYZE to understand query plans; watch for Seq Scan on large tables (needs index)
- Index types: B-Tree (default, equality+range), GIN (full-text, JSONB, arrays), BRIN (very large time-series tables)
- Create partial indexes for common filters (WHERE status='active'); covering indexes (INCLUDE) for index-only scans
- Avoid SELECT * -- fetch only needed columns; avoid functions in WHERE clause (prevents index usage unless functional index exists)
- N+1 query elimination: use JOINs or batch IN(...) queries instead of per-row subqueries
- Cursor-based pagination (WHERE created_at < last_cursor) instead of OFFSET on large tables -- OFFSET scans and discards rows
- Correlated subqueries: rewrite as JOINs with GROUP BY or window functions for major speedups
- Batch INSERT with executemany/VALUES list + single COMMIT; use COPY for bulk loads in PostgreSQL
- Materialized views for expensive aggregations; refresh concurrently to avoid locks
- Partition large tables by date range for automatic partition pruning on time-based queries
- Monitor: pg_stat_statements for slow queries, pg_stat_user_tables for missing indexes (high seq_scan), pg_stat_user_indexes for unused indexes
- Maintenance: run ANALYZE regularly for up-to-date statistics; VACUUM to reclaim space; REINDEX periodically
