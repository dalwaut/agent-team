FASTAPI ARCHITECTURE STANDARDS

When proposing features that add new API endpoints or services, follow these patterns:

- **Project structure**: `api/v1/endpoints/`, `core/` (config, security, db), `models/`, `schemas/`, `services/`, `repositories/`
- **Repository pattern**: DB access only through a repository class (`get`, `get_multi`, `create`, `update`, `delete`) — never raw DB calls in route handlers
- **Service layer**: Business logic belongs in a service class, not in the endpoint function
- **Async all the way**: Never `open()`, `json.load()`, `requests.get()`, or `subprocess.run()` in an `async def` route — use `asyncio.to_thread()` for blocking calls
- **Dependency injection**: Auth, DB session, rate limiting — all via `Depends()`; never global state in handlers
- **Pydantic schemas**: Request and response bodies always validated via Pydantic models — no raw `dict` or `request.json()`
- **Consistent HTTP semantics**: POST (create, 201), GET (read, 200), PATCH (partial update, 200), DELETE (no content, 204) — PUT only for full replacement
- **Common pitfalls to avoid in proposals**:
  - Business logic directly in route functions (belongs in service layer)
  - Missing type hints on function signatures
  - Tight coupling — routes directly importing DB models

## Enhanced Knowledge: Security Requirement Extraction [security-scanning/security-requirement-extraction]
- Every security requirement must trace back to a specific threat or compliance control
- Requirements have three types: Functional (what), Non-functional (how), Constraint (limits)
- Use STRIDE mapping: Spoofing->Authentication, Tampering->Input Validation, Repudiation->Audit Logging, Info Disclosure->Data Protection, DoS->Availability, EoP->Authorization
- Every requirement needs testable acceptance criteria — "be secure" is not a requirement
- Priority = Impact x Likelihood; Critical >= 12, High >= 6, Medium >= 3
- Generate traceability matrix: threat-ID -> requirement-IDs for audit coverage
- Compliance mapping: map requirements to PCI-DSS, HIPAA, GDPR, OWASP ASVS controls
- Run gap analysis per framework to find missing or weakly covered controls
- Convert requirements to security user stories with Definition of Done including security review
- Always include rationale (the "why") — requirements without rationale get deprioritized

## Enhanced Knowledge: FastAPI Templates [api-scaffolding/fastapi-templates]
- Project layout: `api/v1/endpoints/`, `core/` (config, security, db), `models/`, `schemas/`, `services/`, `repositories/`, `main.py`
- Use `@asynccontextmanager` lifespan for startup/shutdown (DB connect/disconnect)
- Dependency injection via `Depends()`: DB sessions, auth, rate limits — never global state
- Repository pattern with generics: `BaseRepository[ModelType, CreateSchema, UpdateSchema]` for CRUD
- Service layer owns business logic; routes only handle HTTP concerns
- Pydantic `BaseSettings` with `@lru_cache` for config; schemas for all request/response bodies
- Async SQLAlchemy sessions: `AsyncSession` with `yield`, commit on success, rollback on exception
- Auth: JWT via `OAuth2PasswordBearer`, `jose` for token encode/decode, `passlib` for bcrypt
- Testing: override `get_db` with in-memory SQLite, `httpx.AsyncClient` for endpoint tests
- Common pitfalls: blocking code in async handlers, business logic in routes, missing type hints, tight DB coupling

## Enhanced Knowledge: API Design Principles [backend-development/api-design-principles]
- Resources are nouns (`/users`), not verbs (`/createUser`); use HTTP methods for actions
- HTTP semantics: GET (read, idempotent), POST (create, 201), PUT (replace), PATCH (partial), DELETE (204)
- Always paginate collections: `page`, `page_size`, `total`, `pages` in response envelope
- Consistent error response format: `{ error, message, details, timestamp, path }`
- Use standard status codes correctly: 400 bad input, 401 unauthed, 403 forbidden, 404 not found, 409 conflict, 422 validation
- Version APIs from day one (URL prefix `/api/v1/` preferred)
- GraphQL: schema-first design, cursor-based pagination (Relay spec), DataLoaders to prevent N+1
- Mutation payloads return `{ data, errors }` for partial success reporting
- HATEOAS: include `_links` for discoverability when relevant
- Rate limit all public endpoints; stricter limits on auth endpoints
- API structure should NOT mirror database schema — design for consumer usability

## Enhanced Knowledge: Architecture Patterns [backend-development/architecture-patterns]
- Clean Architecture: dependencies point inward (Entities <- Use Cases <- Adapters <- Frameworks)
- Inner layers know nothing about outer layers — business logic independent of frameworks
- Hexagonal (Ports & Adapters): domain core defines Ports (interfaces), adapters implement them
- Swap adapters for testing (MockPaymentAdapter) without changing domain logic
- DDD tactical patterns: Entities (identity), Value Objects (immutable), Aggregates (consistency boundary), Domain Events
- Use Cases orchestrate business logic; Controllers handle HTTP concerns only
- Repository pattern abstracts data access — never expose ORM objects to domain layer
- Avoid anemic domain models (entities with only data, no behavior)
- Over-engineering warning: don't use Clean Architecture for simple CRUD
- Bounded Contexts define clear domain boundaries; Ubiquitous Language ensures consistent terminology
- Rich domain models: put business rules and state transitions inside entities

## Enhanced Knowledge: TypeScript Advanced Types [javascript-typescript/typescript-advanced-types]
- Use `unknown` over `any` — forces explicit type checking before use
- Generic constraints: `<T extends HasLength>` to enforce shape requirements
- Conditional types: `T extends string ? A : B` for type-level branching
- `infer` keyword to extract inner types: `T extends Promise<infer U> ? U : never`
- Mapped types for transformations: `{ [K in keyof T]: ... }` with key remapping via `as`
- Template literal types: `on${Capitalize<EventName>}` for string pattern types
- Discriminated unions with `type` field enable exhaustive `switch` narrowing
- Utility types: `Partial`, `Required`, `Pick`, `Omit`, `Record`, `Exclude`, `Extract`, `NonNullable`
- Type guards (`value is string`) and assertion functions (`asserts value is string`) for runtime narrowing
- Deep recursive types: `DeepReadonly<T>`, `DeepPartial<T>` for nested immutability/optionality
- Prefer `interface` for object shapes (better error messages), `type` for unions and complex logic
- Avoid deeply nested conditional types — they slow compilation
