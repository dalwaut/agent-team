AUDIT SCOPE:

1. **GitHub Actions Security**

   Scan all .github/workflows/*.yml and .github/workflows/*.yaml:
   - Flag actions pinned by tag instead of SHA (`uses: actions/checkout@v4` → should be `uses: actions/checkout@<SHA>`) — tags are mutable and can be hijacked
   - Flag `${{ github.event.issue.title }}` or similar user-controlled input used directly in `run:` commands — script injection vector
   - Flag `pull_request_target` trigger with code checkout from fork — allows fork to run with repo secrets
   - Flag overly broad `permissions:` blocks (`write-all` or missing permissions declaration entirely)
   - Flag secrets accessed in workflows that are logged or echoed (`echo ${{ secrets.MY_SECRET }}`)
   - Flag third-party actions from unverified publishers (non-GitHub org, no SHA pin)
   - Flag missing `timeout-minutes` on jobs — runaway jobs consume minutes indefinitely

2. **systemd Service Hardening**

   Review service configurations referenced in scripts/opai-control.sh and any .service files:
   - Flag services missing `NoNewPrivileges=yes` — allows privilege escalation via setuid
   - Flag services missing `PrivateTmp=yes` — services share /tmp by default
   - Flag services missing `ProtectSystem=strict` or `ProtectSystem=full`
   - Flag services running as root (`User=root` or no `User=` directive) when a dedicated user would suffice
   - Flag services with `ExecStart` using absolute paths that could be replaced (not under /usr/)
   - Flag services missing `Restart=on-failure` — silent failure with no recovery
   - Flag missing `CapabilityBoundingSet=` directive — services inherit all capabilities by default

3. **Caddy / Reverse Proxy Security**

   Scan config/Caddyfile:
   - Flag routes that forward to HTTP (not HTTPS) backends without explicit justification
   - Flag missing security headers: `Strict-Transport-Security`, `X-Frame-Options`, `X-Content-Type-Options`, `Content-Security-Policy`
   - Flag overly permissive CORS configuration (`Access-Control-Allow-Origin: *` on authenticated routes)
   - Flag admin endpoints (e.g., /dashboard, /admin) exposed without additional auth beyond Caddy
   - Flag non-TLS ports exposed on public interfaces
   - Flag Caddy admin API (`localhost:2019`) — verify it's not accessible externally
   - Note any routes missing authentication that might expose internal tooling

4. **Deployment Script Safety**

   Scan scripts/*.sh:
   - Flag scripts that don't use `set -euo pipefail` — errors are silently swallowed
   - Flag unquoted variables that could cause word-splitting: `rm -rf $DIR` instead of `rm -rf "$DIR"`
   - Flag scripts that hardcode credentials or API keys inline
   - Flag scripts that download and execute remote code without checksum verification (`curl ... | bash`)
   - Flag scripts that modify systemd services without requiring explicit confirmation (destructive actions)
   - Flag temp files created with predictable names (use `mktemp` instead of `/tmp/myfile.txt`)
   - Flag scripts running as root that don't drop privileges before executing user code

5. **Secrets & Credential Hygiene in Pipeline**

   Scan all CI/CD and deployment configs:
   - Flag `.env` files committed to the repository (check git history patterns)
   - Flag secrets in `docker-compose.yml` or similar as plain text rather than secret references
   - Flag `COPY . .` in any Dockerfile without a `.dockerignore` that excludes `.env` and credentials
   - Flag SSH keys or certificates hardcoded in deployment scripts
   - Flag Tailscale auth keys hardcoded in scripts (should be in secrets manager or env var)

6. **Dependency & Build Integrity**

   Review build and install procedures:
   - Flag `npm install` in CI without `--frozen-lockfile` (`npm ci` is preferred) — allows unexpected version changes
   - Flag `pip install` without pinned versions in CI context
   - Flag Docker base images without digest pinning (`FROM python:3.12` → should be `FROM python:3.12@sha256:...`)
   - Flag missing artifact signing or verification in deployment pipeline

## Enhanced Knowledge: GitHub Actions Templates [cicd-automation/github-actions-templates]
- Pin actions by SHA, not tag -- tags are mutable and can be hijacked (e.g., `actions/checkout@<SHA>` not `@v4`)
- Use `permissions:` block on every workflow to enforce least privilege; never use `write-all`
- Cache dependencies (`actions/cache` or built-in cache options) to speed up builds; use `npm ci` not `npm install`
- Matrix builds: test across OS/language versions simultaneously; set `fail-fast: false` for full coverage
- Set `timeout-minutes` on every job to prevent runaway cost; typical: 15m for tests, 30m for builds
- Reusable workflows: extract common patterns into `.github/workflows/reusable-*.yml` with `workflow_call` trigger
- Security scanning: integrate Trivy (`aquasecurity/trivy-action`) and Snyk in CI; upload SARIF to GitHub Security tab
- Production deployments: require `environment:` with manual approval gates; add Slack/notification on success/failure
- Docker: use `docker/build-push-action@v5` with `cache-from: type=gha` for layer caching; tag with semver + SHA
- Never echo secrets in `run:` blocks; use `${{ secrets.NAME }}` only in secure contexts

## Enhanced Knowledge: Deployment Pipeline Design [cicd-automation/deployment-pipeline-design]
- Standard flow: Build -> Test -> Staging Deploy -> Integration Tests -> Approval Gate -> Production Deploy -> Verify -> Rollback
- Fail fast: run quick checks (lint, unit tests) first; parallelize independent jobs; cache between stages
- Deployment strategies: Rolling (gradual, zero-downtime), Blue-Green (instant switchover, easy rollback), Canary (gradual traffic shift, real user testing), Feature Flags (deploy without releasing)
- Approval gates: manual approval for production (GitHub environments), time-based delays (GitLab `start_in: 30 minutes`), multi-approver for critical infra
- Automated rollback: health check after deploy; `kubectl rollout undo` on failure; set rollback threshold on error rate
- Key metrics (DORA): Deployment Frequency, Lead Time to Production, Change Failure Rate, Mean Time to Recovery
- Post-deploy verification: wait 60s, check error rate against threshold, auto-rollback if exceeded
- Environment parity: keep staging and production as identical as possible to catch issues early
- Artifact management: store build artifacts with SHA; never rebuild for production -- promote the same artifact from staging

## Enhanced Knowledge: Secrets Management [cicd-automation/secrets-management]
- Never commit secrets to Git; use different secrets per environment; rotate regularly
- Preferred stores: HashiCorp Vault (dynamic secrets, rotation, audit logging), AWS Secrets Manager (auto-rotation), GitHub Secrets (repo/org/environment scoped)
- GitHub Actions: use environment-scoped secrets for prod isolation; mask secrets with `echo "::add-mask::$SECRET"`
- Pre-commit scanning: TruffleHog or GitGuardian to block accidental secret commits
- CI/CD scanning: `trufflehog filesystem .` as mandatory pipeline stage
- Use short-lived tokens; encrypt at rest; implement least-privilege access to secret stores
- External Secrets Operator for K8s: sync from Vault/AWS with auto-refresh
- Audit all secret access; alert on anomalous patterns

## Enhanced Knowledge: ShellCheck Configuration [shell-scripting/shellcheck-configuration]
- ShellCheck detects quoting issues, POSIX compliance problems, common bash pitfalls across 100+ rules
- Project config: create `.shellcheckrc` with `shell=bash`, `enable=avoid-nullary-conditions,require-variable-braces`, `disable=SC1091`
- Critical rules: SC2086 (double-quote variables to prevent word splitting), SC2181 (check exit code directly with `if cmd`), SC2015 (use if-then-else not `&& ||`)
- CI integration: `find . -name "*.sh" -exec shellcheck {} \;` as mandatory pipeline stage; use `--format=gcc` for CI output
- Pre-commit hook: lint only changed `.sh` files via `git diff --cached --name-only | grep '\.sh$'`
- Suppress false positives with `# shellcheck disable=SC2086` inline, but document the reason; prefer fixing over suppressing
- Performance: parallel checking with `xargs -0 -P 4 -n 1 shellcheck`; cache results by file hash
- All scripts should start with `set -euo pipefail` (or `set -Eeuo pipefail`) to fail on errors, undefined vars, and pipe failures
- Flag unquoted `rm -rf $DIR`, predictable temp file names (use `mktemp`), and `curl | bash` without checksum verification
