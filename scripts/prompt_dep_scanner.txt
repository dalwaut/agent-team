CHECKS PER MANIFEST:

1. **Unpinned versions** (Python)
   - Flag any package without an exact version pin (`requests` with no `==x.y.z`)
   - Flag ranges `>=`, `~=`, `>` — these allow unexpected upgrades
   - Flag `latest` or `*` in any manifest format
   - Ideal: every prod dependency pinned to exact version; dev deps can use ranges

2. **Overly permissive Node.js ranges**
   - Flag `*` or `latest` — no version constraint at all
   - Flag `^` (caret) on major version 0 packages (0.x.y) — minor bump = breaking change
   - Flag missing `package-lock.json` or `yarn.lock` alongside `package.json`
   - Flag `dependencies` that should be `devDependencies` (test frameworks, linters, type-defs)

3. **Known vulnerable version patterns** (flag by name + version range — verify against common advisories):
   - Python: `requests <2.31`, `urllib3 <2.0`, `pillow <10.0`, `pyyaml <6.0`, `cryptography <41`, `django <4.2`, `flask <3.0`, `jinja2 <3.1`, `paramiko <3.0`, `setuptools <65`
   - Node.js: `axios <1.6`, `lodash <4.17.21`, `minimist <1.2.6`, `node-fetch <2.6.7`, `express <4.18`, `jsonwebtoken <9.0`, `semver <7.5.2`, `tough-cookie <4.1.3`
   - Flag any package version that is 2+ major versions behind current (research from package manifest metadata)

4. **Lifecycle scripts risk** (Node.js)
   - Flag `postinstall`, `preinstall`, `install` scripts in `package.json` — these run arbitrary code on `npm install`
   - Check if the script is from a trusted own-code path vs. a third-party dep script

5. **Missing lock files**
   - Flag any `package.json` without a corresponding lock file in the same directory
   - Flag any `requirements.txt` without a `requirements.lock` or pip-compile output (note as advisory)

6. **Duplicate packages** (Node.js)
   - In `package-lock.json`, flag the same package appearing at multiple version levels (version conflicts)

7. **Dev dependencies in production manifests**
   - Flag test frameworks (`pytest`, `jest`, `mocha`), linters (`eslint`, `flake8`), type stubs in production requirements

## Enhanced Knowledge: Dependency Upgrade [framework-migration/dependency-upgrade]
- Semver ranges: `^2.3.1` = >=2.3.1 <3.0.0 (minor), `~2.3.1` = >=2.3.1 <2.4.0 (patch), exact pin = safest for prod
- Audit commands: `npm outdated`, `npm audit`, `npx npm-check-updates`; analyze tree with `npm ls package-name` or `yarn why`
- Staged upgrade strategy: plan order (least-dependent first), upgrade one major version at a time, test after each change
- Breaking change handling: read CHANGELOG.md and MIGRATION.md, use codemods (`npx jscodeshift`) for automated API migration
- Peer dependency resolution: `npm install --legacy-peer-deps` to bypass strict checks; `--force` for overrides (npm 8+)
- Automated updates: Renovate (automerge minor/patch, label majors) or Dependabot (weekly schedule, PR limits, reviewer assignment)
- Rollback plan: always create feature branch + git tag before upgrade; restore from package-lock.json on failure
- Lock file discipline: `npm ci` for clean installs from lock, never skip lock file commits; `yarn install --frozen-lockfile` in CI
- Upgrade checklist: review versions, read changelogs, create branch, backup state, run baseline tests, upgrade one-at-a-time, fix TS errors, check bundle size, regression test, deploy to staging, monitor
- Common pitfalls: upgrading all deps at once, ignoring peer dep warnings, skipping major versions, not having rollback plan
