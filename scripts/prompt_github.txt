## Enhanced Knowledge: Git Advanced Workflows [developer-essentials/git-advanced-workflows]
- Interactive rebase operations: pick, reword, edit, squash, fixup, drop -- use to clean feature branches before PR
- Cherry-pick: apply specific commits across branches (`git cherry-pick -n` for staging only, `-e` to edit message)
- Git bisect: binary search for bug-introducing commit; automate with `git bisect run ./test.sh` (exit 0=good, 1-127=bad)
- Worktrees: work on multiple branches simultaneously without stashing (`git worktree add ../path branch-name`)
- Reflog: safety net tracking all ref movements for 90 days; recover deleted branches/commits via `git reflog` + `git branch recovered abc123`
- Always use `--force-with-lease` instead of `--force` to prevent overwriting others' work
- Rebase vs Merge: rebase for cleaning local commits and linear history; merge for integrating completed features and public branches
- Autosquash workflow: `git commit --fixup HEAD` then `git rebase -i --autosquash main` for automatic fixup ordering
- Split commits via interactive rebase: mark with 'edit', then `git reset HEAD^` and recommit in logical chunks
- Best practices: atomic commits (one logical change each), backup branch before risky operations, never rebase public branches

## Enhanced Knowledge: Changelog Automation [documentation-generation/changelog-automation]
- Follow Keep a Changelog format with sections: Added, Changed, Deprecated, Removed, Fixed, Security
- Conventional Commits: `<type>[scope]: <description>` -- types map to changelog sections (feat->Added, fix->Fixed, perf->Changed)
- Semantic Versioning: MAJOR (breaking, feat!), MINOR (feat), PATCH (fix) -- enforce via commitlint + husky hooks
- Toolchain options: standard-version (manual release), semantic-release (full CI automation), git-cliff (Rust, fast), commitizen (Python)
- commitlint config: enforce type-enum, subject-case, subject-max-length (72 chars) via `.husky/commit-msg` hook
- GitHub Actions release workflow: checkout with fetch-depth 0, configure git user, run semantic-release with GITHUB_TOKEN
- Breaking changes: mark with `!` suffix on type or `BREAKING CHANGE:` footer -- triggers MAJOR version bump
- Release notes: group by category (Features, Bug Fixes, Docs), reference PR numbers and authors, list new contributors
- Best practices: one logical change per commit, reference issues, use scopes consistently, automate releases to reduce errors
- Never manually edit generated changelogs; validate commits in CI pipeline; always mark breaking changes explicitly
