AUDIT SCOPE:

1. **Container Privilege & User**

   Scan all Dockerfiles:
   - Flag containers with no `USER` directive — default is root, highest risk
   - Flag `USER root` explicitly set (intentional privilege, still flag)
   - Flag `--privileged` flag in docker-compose or run commands — grants full host access
   - Flag `cap_add: ALL` or `cap_add: SYS_ADMIN` without clear justification
   - Flag `securityContext.privileged: true` in any k8s-adjacent configs

2. **Dangerous Volume & Mount Patterns**

   Scan docker-compose files and run commands:
   - Flag Docker socket mounts: `-v /var/run/docker.sock:/var/run/docker.sock` — container with socket has full Docker daemon access = container escape
   - Flag host path mounts of sensitive directories: `/etc`, `/root`, `/home`, `/proc`, `/sys`
   - Flag read-write mounts of paths that should be read-only (config files, static assets)
   - Flag named volumes without explicit driver/backup strategy for stateful data

3. **Secrets in Images**

   Scan Dockerfiles and compose files:
   - Flag `ENV MY_SECRET=value` with literal secret values (baked into image layer, visible in `docker inspect`)
   - Flag `ARG` used to pass secrets at build time — visible in build history (`docker history`)
   - Flag `COPY . .` without a `.dockerignore` that excludes `.env`, `*.key`, `credentials*`, `notes/Access/`
   - Flag hardcoded URLs with embedded credentials in environment variables
   - Flag `RUN` commands that `curl` or `wget` with auth headers containing tokens

4. **Base Image Hygiene**

   Scan all `FROM` directives:
   - Flag `FROM <image>:latest` — non-deterministic, can pull breaking changes
   - Flag base images that haven't been updated in the Dockerfile (no recent `apt-get update && apt-get upgrade`)
   - Flag use of unofficial/unverified base images (prefer `docker.io/library/` official images or digests)
   - Flag missing digest pinning: `FROM python:3.12` → recommend `FROM python:3.12@sha256:...`
   - Flag base images with known critical CVEs (check image name against common advisory patterns)

5. **Resource Limits**

   Scan docker-compose files and any orchestration config:
   - Flag services missing `mem_limit` or `memory:` constraint — container can OOM the host
   - Flag services missing `cpus:` limit — runaway container can starve other services
   - Flag missing `restart:` policy — service failures go unrecovered
   - Flag missing health checks (`healthcheck:`) on long-running services

6. **Network Exposure**

   Scan port mappings and network config:
   - Flag `ports: "0.0.0.0:PORT:PORT"` binding to all interfaces for internal services (use `127.0.0.1:PORT:PORT`)
   - Flag services on the same Docker network that don't need to communicate (violates least privilege)
   - Flag admin interfaces (Portainer, Coolify dashboard, n8n) exposed without additional auth layer
   - Flag missing `--network` isolation — all containers on the default bridge share a network

7. **Runtime Security**

   Scan compose files and run commands:
   - Flag `security_opt: []` explicitly disabling seccomp/apparmor profiles
   - Flag missing `read_only: true` on container filesystem where possible
   - Flag `pid: host` — shares host PID namespace, allows process inspection/signaling
   - Flag `network_mode: host` — bypasses Docker network isolation entirely
   - Flag containers with write access to paths they only need to read

## Enhanced Knowledge: Kubernetes Security Policies [kubernetes-operations/k8s-security-policies]
- Pod Security Standards (namespace labels): Privileged (unrestricted), Baseline (minimal restrictions), Restricted (hardened -- enforce this for production)
- NetworkPolicy: start with default-deny-all (ingress + egress), then allow specific pod-to-pod communication by label selector and port
- Always allow DNS egress (UDP 53 to kube-system) or pods cannot resolve service names
- RBAC: use namespace-scoped Roles (not ClusterRoles) where possible; bind to ServiceAccounts, not users; follow least privilege
- Pod securityContext checklist: runAsNonRoot: true, runAsUser: 1000, allowPrivilegeEscalation: false, readOnlyRootFilesystem: true, drop ALL capabilities, seccompProfile: RuntimeDefault
- OPA Gatekeeper / Kyverno: enforce admission policies (required labels, no privileged containers, image registry whitelist)
- Service mesh (Istio): enable STRICT mTLS via PeerAuthentication; use AuthorizationPolicy to restrict service-to-service access by ServiceAccount
- CIS Kubernetes Benchmark: enable audit logging, secrets encryption at rest, node authentication, network policies
- Troubleshooting: verify CNI supports NetworkPolicy; check permissions with `kubectl auth can-i list pods --as system:serviceaccount:ns:sa`
- Resource controls: implement ResourceQuotas and LimitRanges per namespace to prevent resource exhaustion
- Regular image scanning: scan all container images for CVEs before deployment and on schedule
