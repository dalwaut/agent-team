AUDIT SCOPE:

1. **REST Naming Conventions**

   Scan all route definitions in routes_*.py files:
   - Flag routes using verbs in the URL path: `/getUser`, `/createTask`, `/deleteItem` — use nouns + HTTP methods instead
   - Flag inconsistent noun plurality: mix of `/user` and `/tasks` — pick one convention (plural preferred)
   - Flag snake_case vs kebab-case inconsistency in path segments: `/user_profile` vs `/user-profile`
   - Flag nested resource paths deeper than 3 levels: `/a/b/c/d` — consider flattening or using query params
   - Flag trailing slashes on routes: `/api/tasks/` — inconsistent trailing slash causes 301 redirects

2. **HTTP Method Correctness**

   Review every route's HTTP method choice:
   - Flag GET routes with side effects (modifying DB, sending emails, running jobs) — use POST/PATCH
   - Flag POST routes used purely for retrieval (should be GET with query params, or GET + request body for complex filters)
   - Flag DELETE routes that return response bodies — 204 No Content is the standard
   - Flag PUT used for partial updates — PUT implies full resource replacement; use PATCH for partial
   - Flag state changes triggered by GET requests (idempotency violation)

3. **Status Code Consistency**

   Scan all endpoint return statements and HTTPException raises:
   - Flag: resource creation returning 200 instead of 201
   - Flag: validation errors returning 500 instead of 422 (Pydantic handles this automatically — check for manual overrides)
   - Flag: not found cases returning 200 with empty body instead of 404
   - Flag: auth failures returning 400 instead of 401 (unauthenticated) or 403 (unauthorized)
   - Flag: 200 returned with `{"success": false, "error": "..."}` in body — use proper HTTP status codes
   - Flag: inconsistent error response schema — some endpoints return `{"detail": "..."}` (FastAPI default), others return `{"error": "...", "message": "..."}` (custom)

4. **Pagination on List Endpoints**

   Scan all GET endpoints that return collections:
   - Flag list endpoints with no `limit`/`offset` or cursor-based pagination
   - Flag list endpoints with a hardcoded limit that isn't surfaced as a query param
   - Flag missing `total` count in paginated responses (clients can't build pagination UI without it)
   - Flag inconsistent pagination shape: some return `{"items": [], "total": N}`, others return bare arrays

5. **Request Validation**

   Scan route function signatures and Pydantic models:
   - Flag route handlers accepting `dict` or `Any` instead of typed Pydantic models for request bodies
   - Flag path/query params with no type annotation — FastAPI can't validate untyped params
   - Flag string fields with no `max_length` constraint (unbounded string inputs at API boundary)
   - Flag missing `Optional` on fields that the API silently treats as optional
   - Flag file upload endpoints with no `content_type` or `max_size` validation

6. **API Versioning**

   Review the URL structure:
   - Flag mix of versioned (`/api/v1/`) and unversioned (`/api/`) routes in the same tool
   - Flag absence of any versioning strategy — note tools most at risk for mobile app breakage (changes break the app until users update)
   - Cross-reference docs/mobile-api-reference.md: flag routes listed there that don't exist or have different signatures

7. **Error Response Standardization**

   Review error handling patterns:
   - Flag routes with no exception handling that could leak stack traces
   - Flag tools with no global exception handler (FastAPI `@app.exception_handler`)
   - Document the current error schema per tool — identify the N different formats in use
   - Flag endpoints returning HTML error pages (Caddy/uvicorn defaults) instead of JSON

## Enhanced Knowledge: API Design Principles [backend-development/api-design-principles]
- Resources are nouns (users, orders, products), not verbs; use HTTP methods for actions; URLs represent resource hierarchies
- HTTP method semantics: GET (retrieve, idempotent, safe), POST (create), PUT (replace entire resource, idempotent), PATCH (partial update), DELETE (remove, idempotent)
- Consistent naming: plural nouns for collections (`/users` not `/user`); avoid action-oriented endpoints (`/createUser`)
- Pagination: always paginate large collections; include page, page_size, total, pages, has_next, has_prev in response
- Error handling: use proper HTTP status codes (201 for creation, 404 for not found, 422 for validation); standardize error response schema with error code, message, details
- HATEOAS: include `_links` with self, related resources, and available actions in responses for API discoverability
- GraphQL patterns: schema-first development, use DataLoaders to prevent N+1, cursor-based pagination (Relay spec), structured errors in mutation payloads
- API versioning: plan from day one; URL versioning (`/api/v1/`) is most common; header versioning for more flexibility
- Rate limiting: protect all APIs; document limits in headers (X-RateLimit-Limit, X-RateLimit-Remaining, Retry-After)
- Common pitfalls: breaking changes without versioning, inconsistent error formats, missing rate limits, tight coupling to DB schema, ignoring HTTP method semantics

## Enhanced Knowledge: OpenAPI Spec Generation [documentation-generation/openapi-spec-generation]
- OpenAPI 3.1 structure: info, servers, paths, components (schemas, securitySchemes, parameters, responses, examples)
- Design approaches: Design-First (spec before code, best for new APIs/contracts), Code-First (generate from code, best for existing APIs), Hybrid (annotated code)
- Use `$ref` to reuse schemas, parameters, and responses; add real-world examples to all endpoints
- Every endpoint needs: operationId, summary, description, tags, responses (including error codes), security scheme
- Define reusable error schemas with code, message, details array, and requestId fields
- Pagination components: page, limit, total, totalPages, hasNext, hasPrev — standardize across all endpoints
- Lint specs with Spectral (`spectral lint openapi.yaml`) and Redocly for validation; enforce naming conventions (camelCase properties, snake_case path params)
- Generate client SDKs with openapi-generator-cli for TypeScript, Python, Go
- Document all security schemes (bearerAuth, apiKey) and rate limiting (Retry-After, X-RateLimit-Limit headers)
- Never use generic descriptions, never skip security definitions, never hardcode URLs — use server variables
