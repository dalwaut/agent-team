PERFORMANCE — Focus on:

1. Database/API queries: unbounded queries (no LIMIT), N+1 patterns, redundant network calls, missing pagination
2. Frontend rendering: unnecessary re-renders, expensive computations inside render cycles, missing memoization
3. Bundle size: heavy dependencies that could be replaced with lighter alternatives, tree-shaking opportunities
4. Memory: event listeners not cleaned up, intervals not cleared, subscriptions not unsubscribed

5. **Python/FastAPI async blocking** (CRITICAL — a single blocking call stalls the entire event loop):
   - Flag `time.sleep()` inside any `async def` — require `await asyncio.sleep()`
   - Flag `requests.get/post/put/delete()` inside any `async def` — require `httpx.AsyncClient` with `await`
   - Flag synchronous `open()` / file I/O inside any `async def` — require `asyncio.to_thread()` or `aiofiles`
   - Flag `subprocess.run()` / `subprocess.call()` inside any `async def` — require `asyncio.create_subprocess_exec()` or `asyncio.to_thread()`
   - Flag `psycopg2` usage (blocking) where `asyncpg` or `psycopg3` async driver is available
   - Flag missing `asyncio.Semaphore` on routes that spawn unbounded parallel tasks (e.g., `asyncio.gather(*[...])` over user-supplied lists)

6. **Node.js async patterns** (for .js files in tools/):
   - Flag synchronous `fs.readFileSync` / `fs.writeFileSync` inside request handlers — use `fs.promises` or streams
   - Flag unhandled promise rejections (`.then()` without `.catch()`, or `async` functions called without `await` or `.catch()`)
   - Flag `setInterval` / `setTimeout` handlers that are never cleared on shutdown
   - Flag missing `process.on('uncaughtException')` and `process.on('unhandledRejection')` handlers in long-running server processes

## Enhanced Knowledge: Parallel Debugging [agent-teams/parallel-debugging]
- Generate hypotheses across 6 failure mode categories: Logic Error, Data Issue, State Problem, Integration Failure, Resource Issue, Environment
- Logic errors: wrong conditional operator, off-by-one, missing edge case, incorrect algorithm
- Data issues: invalid input, type mismatch, null where value expected, encoding/serialization problem, truncation/overflow
- State problems: race condition, stale cache, incorrect initialization, unintended mutation of shared state, state machine transition error
- Integration failures: API contract violation, version incompatibility, config mismatch, missing env vars, network timeout
- Resource issues: memory leak, connection pool exhaustion, file descriptor leak, disk quota exceeded, CPU saturation
- Evidence strength ranking: Direct (code at file:line) > Correlational (error rate after commit) > Testimonial > Absence
- Confidence levels: High (>80%) = multiple direct evidence + clear causal chain; Medium (50-80%) = some direct evidence; Low (<50%) = mostly correlational
- Arbitration: categorize as Confirmed/Plausible/Falsified/Inconclusive; rank confirmed by confidence, evidence count, causal chain strength
- Validation checklist: fix addresses root cause, no new issues introduced, original repro passes, edge cases covered, tests added

## Enhanced Knowledge: Debugging Strategies [developer-essentials/debugging-strategies]
- Follow the scientific method: Observe -> Hypothesize -> Experiment -> Analyze -> Repeat
- Phase 1 Reproduce: create minimal reproduction, document exact steps, note environment details
- Phase 2 Gather Info: full stack trace, error codes, environment versions, recent git changes, deployment timeline, affected scope
- Phase 3 Hypothesize: ask "What changed?", "What is different?" (working vs broken env/user/time), "Where could this fail?"
- Phase 4 Test: binary search (comment out half), add strategic logging, isolate components, compare working vs broken via diff
- Use git bisect to pinpoint regression commits; differential debugging table (Aspect | Working | Broken)
- Intermittent bugs: add timing logs, look for race conditions, check async ordering, stress test under load
- Memory leak detection: take heap snapshots before/after, compare, track growth per operation
- Quick stuck checklist: typos, case sensitivity, null/undefined, off-by-one, async timing, scope issues, type mismatches, env vars, file paths, cache staleness
- Common mistakes: making multiple changes at once, not reading full stack trace, assuming complexity, leaving debug logging in production
