CHECKS:

1. **Authentication Coverage**

   For every endpoint:
   - Classify as: PUBLIC (no auth needed), USER (requires valid session), ADMIN (requires admin role)
   - Flag any endpoint that performs write operations (POST, PUT, PATCH, DELETE) without an auth dependency or middleware check
   - Flag any endpoint accessing user-specific data (by user_id, by session) without verifying the requesting user owns that data
   - Flag any admin-only operation (agent execution, config changes, user management) without an explicit admin role check
   - In FastAPI: check for `Depends(get_current_user)` or equivalent on protected routes
   - In Node.js: check for auth middleware called before the route handler

2. **HTTP Method Correctness**

   - Flag GET endpoints that create, modify, or delete data (GET must be idempotent and side-effect free)
   - Flag POST endpoints used for data retrieval that should be GET (unless body params make GET impractical)
   - Flag missing PATCH vs PUT distinction — PATCH = partial update, PUT = full replacement
   - Flag DELETE endpoints that don't actually delete (soft-delete without documentation)

3. **Request Validation**

   For every POST, PUT, and PATCH endpoint:
   - Flag missing Pydantic model for request body (FastAPI) — bare `dict` or `request.json()` without validation
   - Flag missing type annotations on path parameters (`:id` with no type = accepts anything)
   - Flag missing length limits on string inputs (can cause DoS with massive payloads)
   - Flag file upload endpoints without file type and size validation
   - In Node.js: flag `req.body.field` usage without express-validator or joi schema validation

4. **Response Format Consistency**

   Sample 10+ endpoints and check:
   - Is the success response format consistent? (`{"data": ...}` vs `{"result": ...}` vs bare object)
   - Is the error response format consistent? (`{"error": "message"}` vs `{"detail": "message"}` vs `{"message": "..."}`)
   - Flag endpoints returning HTTP 200 for errors (should use 4xx/5xx status codes)
   - Flag endpoints returning raw exceptions or stack traces in production error responses
   - Flag missing Content-Type headers on responses

5. **Rate Limiting**

   Flag endpoints missing rate limiting that are abuse-prone:
   - Authentication endpoints (login, signup, password reset, token refresh)
   - Email-sending endpoints
   - Search endpoints (expensive DB queries)
   - File upload or generation endpoints
   - Any endpoint callable without authentication (public API surface)
   - Check for existing rate limit middleware — if present, is it applied globally or per-route?

6. **CORS and Security Headers**

   - Flag overly permissive CORS: `Access-Control-Allow-Origin: *` on endpoints that serve authenticated data
   - Flag missing CORS configuration entirely on API services (defaults may be insecure)
   - Flag missing security headers: `X-Content-Type-Options`, `X-Frame-Options`, `Referrer-Policy`
   - Flag endpoints that echo user input in response headers (header injection risk)

7. **API Versioning**

   - Is there an API versioning strategy? (`/api/v1/`, `/v1/`, custom header versioning)
   - Flag APIs with no versioning (breaking changes will immediately break all clients)
   - Flag mixed versioning patterns across different tools (some use `/v1/`, some don't)

8. **Async Correctness** (FastAPI-specific)

   - Flag `async def` route handlers that call blocking sync functions (already covered by health agent — cross-reference only)
   - Flag `def` (sync) route handlers doing I/O-heavy operations — should be `async def` in FastAPI

## Enhanced Knowledge: API Design Principles [backend-development/api-design-principles]
- Resources are nouns (users, orders, products), not verbs; use HTTP methods for actions; URLs represent resource hierarchies
- HTTP method semantics: GET (retrieve, idempotent, safe), POST (create), PUT (replace entire resource, idempotent), PATCH (partial update), DELETE (remove, idempotent)
- Consistent naming: plural nouns for collections (`/users` not `/user`); avoid action-oriented endpoints (`/createUser`); pick one convention for snake_case vs kebab-case
- Pagination: always paginate large collections; cursor-based pagination preferred over OFFSET for large datasets; include total count for UI
- Error handling: use proper HTTP status codes (201 created, 204 no content on delete, 400 bad request, 401 unauthenticated, 403 unauthorized, 404 not found, 409 conflict, 422 validation error)
- Standardize error response schema: `{"error": "ErrorCode", "message": "Human-readable", "details": [...], "requestId": "..."}` — never return 200 with `{"success": false}`
- GraphQL: schema-first, DataLoaders to prevent N+1, cursor-based Relay pagination, `@deprecated` directive for migration, structured errors in mutation payloads
- Versioning: plan from day one; URL versioning (`/api/v1/`) most common; breaking changes without versioning immediately break all clients
- Rate limiting: protect auth endpoints, search endpoints, file uploads, and any public surface; document limits in response headers
- Stateless design: each request contains all necessary information; never rely on server-side session state for REST APIs
