## Enhanced Knowledge: STRIDE Analysis Patterns [security-scanning/stride-analysis-patterns]
- STRIDE categories: Spoofing (AuthN), Tampering (integrity), Repudiation (logging), Info Disclosure (confidentiality), DoS (availability), Elevation (AuthZ)
- Per-element threat mapping: External entities -> S,R; Processes -> all six; Data stores -> T,R,I,D; Data flows -> T,I,D
- Always identify trust boundaries first (Internet->DMZ, DMZ->Internal, App->DB), then analyze each crossing
- Risk score = Impact x Likelihood (1-4 scale each); Critical >= 12, High >= 6, Medium >= 3
- Key mitigations: S -> MFA + secure tokens; T -> input validation + parameterized queries; R -> tamper-evident audit logs; I -> encryption at rest/transit; D -> rate limiting + auto-scaling; E -> server-side RBAC + least privilege
- Find unencrypted data flows crossing trust boundaries -- these are high-priority gaps
- STRIDE per Interaction: apply STRIDE to each (source_type, target_type) pair, not just components

## Enhanced Knowledge: Attack Tree Construction [security-scanning/attack-tree-construction]
- Root goal = attacker objective; decompose into sub-goals via OR (any child) and AND (all children required) nodes; leaves are atomic attack steps
- Attribute each leaf: cost, time, skill level, detection risk, requires_insider, requires_physical
- OR node difficulty = min(children); AND node difficulty = max(children); OR cost = min; AND cost = sum
- Find easiest/cheapest/stealthiest paths to prioritize defenses -- block the lowest-cost attack path first
- Critical nodes = nodes appearing in the most attack paths; mitigating these gives highest coverage
- Coverage analysis: given mitigated attacks, calculate what % of paths are blocked vs. still open
- Unmitigated leaf nodes (no mitigations listed) are the highest-priority gaps
- Export to Mermaid/PlantUML for stakeholder communication; color-code by difficulty

## Enhanced Knowledge: Threat Mitigation Mapping [security-scanning/threat-mitigation-mapping]
- Control types: Preventive (stop attack), Detective (identify in progress), Corrective (respond/recover)
- Control layers: Network (firewall, WAF), Application (auth, validation), Data (encryption, ACL), Endpoint (EDR), Process (training, IR)
- Defense in depth: every threat needs controls at 2+ layers AND 2+ control types (preventive + detective minimum)
- Coverage score = effectiveness x implementation_status; flag any threat with < 50% coverage
- Standard control IDs: AUTH-001 (MFA), VAL-001 (input validation), ENC-001 (encryption at rest), LOG-001 (security event logging), ACC-001 (RBAC), AVL-001 (rate limiting)
- Generate remediation roadmap: Phase 1 = critical threats with low coverage, Phase 2 = high-impact threats
- Single controls are single points of failure -- always flag; unmapped threats are coverage 0% and highest priority
- Test control effectiveness regularly; untested controls may silently fail

## Enhanced Knowledge: PCI Compliance [payment-processing/pci-compliance]
- 12 core requirements: secure network, protect cardholder data, vulnerability mgmt, strong access control, monitor/test, security policy
- NEVER store CVV/CVC, full track data, or PIN -- not even temporarily in logs or variables
- PAN must be encrypted at rest (AES-256); mask in logs (first 6 + last 4 only)
- Compliance levels: L1 (>6M txn, annual ROC), L2 (1-6M, SAQ), L3 (20K-1M e-commerce), L4 (<20K e-commerce)
- SAQ A = hosted payment page (least burden); SAQ A-EP = embedded JS form; SAQ D = store/process card data (most burden)
- Reduce scope: use hosted payments, tokenization, network segmentation, outsource to PCI-compliant processors
- Require MFA for access to cardholder data environments; audit log all access with user, action, timestamp, IP
- Common violations: storing CVV, unencrypted PAN, weak encryption, no access controls, missing audit logs, default passwords

## Enhanced Knowledge: Auth Implementation Patterns [developer-essentials/auth-implementation-patterns]
- AuthN (who are you) vs AuthZ (what can you do) -- verify both on every request server-side
- JWT: short-lived access tokens (15m), long-lived refresh tokens (7d) stored hashed in DB; always verify with `jwt.verify()`
- Session-based: use Redis/DB store, httpOnly + secure + sameSite=strict cookies; never expose session ID to JS
- OAuth2/OIDC: delegate auth via Google/GitHub Strategy; generate JWT after OAuth callback, redirect to frontend
- RBAC: define role hierarchy (admin > moderator > user); use middleware `requireRole()` on every protected route
- Permission-based ACL: map roles to granular permissions (read:users, write:posts); check with `requirePermission()`
- Resource ownership: check `resource.userId === req.user.userId` before allowing mutations; admins bypass
- Password security: bcrypt with 12+ rounds, enforce min 12 chars + uppercase + lowercase + number + special
- Rate limit auth endpoints (5 attempts/15min); log all auth events; implement CSRF protection for session-based auth
- Never store JWT in localStorage (XSS risk) -- use httpOnly cookies; rotate secrets regularly

## Enhanced Knowledge: GDPR Data Handling [hr-legal-compliance/gdpr-data-handling]
- Personal data categories: Basic (name, email), Sensitive Art.9 (health, religion), Criminal Art.10, Children's (under 16)
- Six legal bases (Art.6): Consent, Contract, Legal Obligation, Vital Interests, Public Interest, Legitimate Interest
- Data subject rights (must respond within 30 days): Access (Art.15), Rectification (Art.16), Erasure (Art.17), Restrict (Art.18), Portability (Art.20), Object (Art.21)
- Consent must be: freely given, specific, informed, opt-in (never pre-checked); separate consent per purpose
- Data minimization: collect only what is needed for stated purpose; generalize location data to country level
- Privacy by design: separate PII from behavioral data, encrypt PII at rest, use pseudonymized analytics
- Retention policies: define period per data type with legal basis; automate deletion/anonymization after expiry
- Breach notification: notify supervisory authority within 72 hours; notify affected individuals for high-severity breaches
- DSAR automation: implement access, erasure, portability, rectification request handlers; check for legal retention exceptions before erasure
