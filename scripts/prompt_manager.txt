## Enhanced Knowledge: Task Coordination Strategies [agent-teams/task-coordination-strategies]
- Decompose by layer (frontend/backend/DB/tests), by component (auth/profile/notifications), or by file ownership
- Minimize dependency chain depth — prefer wide, shallow graphs over deep sequential chains
- Critical path = longest dependency chain = minimum completion time
- Use `blockedBy` sparingly — only for true dependencies; false dependencies kill parallelism
- Every task description must include: Objective, Owned Files, Requirements, Interface Contracts, Acceptance Criteria, Scope Boundaries
- One agent per file boundary — never assign two tasks that touch the same file
- Workload signals: teammate idle (reassign), teammate stuck (check in), all blocked (resolve critical path first)
- Rebalance via TaskList -> identify idle/overloaded -> TaskUpdate reassign -> SendMessage notify
- Independent tasks (no shared files/state) can run in parallel; sequential tasks need DEPENDS ON markers
- Diamond pattern (`A -> B,C -> D`) maximizes parallelism while respecting convergence points

## Enhanced Knowledge: Team Composition Patterns [agent-teams/team-composition-patterns]
- Team sizing: Simple=1-2, Moderate=2-3, Complex=3-4, Very Complex=4-5 agents
- Start with smallest team covering all required dimensions — each additional agent adds coordination overhead
- Preset teams: Review (3 reviewers), Debug (3 investigators), Feature (1 lead + 2 implementers), Fullstack (1 lead + 3 implementers)
- Agent type selection: `general-purpose` (all tools), `Explore`/`Plan` (read-only), specialized (team-reviewer, team-debugger, team-implementer, team-lead)
- Never assign implementation tasks to read-only agents (Explore, Plan)
- Every team needs a coordinator — either a `team-lead` agent or the user
- Avoid duplicate roles — two agents doing the same thing wastes resources
- Define file/responsibility boundaries upfront before spawning teammates
- Security team preset: 4 reviewers covering OWASP, auth/access, dependencies, secrets
- Migration team: 1 lead + 2 implementers + 1 reviewer for large codebase migrations

## Enhanced Knowledge: Architecture Decision Records [documentation-generation/architecture-decision-records]
- ADR captures: Context (why), Decision (what), Consequences (what happens)
- Write ADRs for: framework adoption, DB tech choice, API patterns, security architecture, integration patterns
- Skip ADRs for: minor version upgrades, bug fixes, implementation details, config changes
- ADR lifecycle: Proposed -> Accepted -> Deprecated -> Superseded (or Rejected)
- Template: Status, Context, Decision Drivers, Considered Options (with pros/cons), Decision, Rationale, Consequences (positive + negative), Implementation Notes
- Lightweight format: Y-statement — "In the context of X, facing Y, we decided Z, accepting W"
- Never modify accepted ADRs — write a new one to supersede with lessons learned
- Include at least 2-3 options with honest trade-offs; always recommend one
- Keep ADRs short (1-2 pages max); link related decisions to build a decision graph
- Review checklist: context clear, options balanced, consequences documented, security/cost considered, reversibility assessed
- Store in `docs/adr/` with numbered files and an index README
