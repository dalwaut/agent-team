## Enhanced Knowledge: Distributed Tracing [observability-monitoring/distributed-tracing]
- Trace structure: Trace (end-to-end request) contains Spans (individual operations) with Context (propagated metadata), Tags, and Logs
- OpenTelemetry is the recommended instrumentation standard; export to Jaeger or Grafana Tempo for storage/visualization
- Context propagation via W3C `traceparent`/`tracestate` HTTP headers; inject context into all outbound requests
- Sampling strategies: Probabilistic (1-10% in prod), Rate-limiting (max N traces/sec), Adaptive (parent-based, deterministic)
- Correlate traces with logs: embed trace_id in structured log entries for cross-referencing
- Jaeger queries for debugging: `service=X duration>1s` for slow requests, `error=true tags.http.status_code>=500` for errors
- Auto-generated service dependency graphs show relationships, request rates, error rates, and latencies
- Best practices: meaningful span tags (user_id, request_id), consistent operation naming, monitor tracing overhead (<1% CPU)
- Troubleshooting: check collector endpoint connectivity, verify sampling config, review batch span processor settings
- Keep tracing overhead minimal; use batch exporters; document instrumentation standards for the team

## Enhanced Knowledge: SLO Implementation [observability-monitoring/slo-implementation]
- Hierarchy: SLA (customer contract) > SLO (internal target) > SLI (actual measurement)
- Common SLIs: Availability (successful/total requests), Latency (requests below threshold/total), Durability (successful writes/total)
- SLO targets: 99% = 7.2hr downtime/mo, 99.9% = 43.2min/mo, 99.95% = 21.6min/mo, 99.99% = 4.32min/mo
- Error Budget = 1 - SLO Target; policy: 100% remaining = normal velocity, 50% = caution, 10% = freeze non-critical, 0% = feature freeze
- Multi-window burn rate alerts reduce false positives: fast burn (14.4x rate, 1hr window) + slow burn (6x rate, 6hr window)
- Prometheus recording rules: pre-compute SLI ratios, SLO compliance booleans, error budget remaining %, burn rates
- Dashboard essentials: current SLO compliance, error budget remaining (gauge), SLI trend (28-day), burn rate analysis
- Review cadence: weekly (compliance, budget, trends), monthly (achievement, postmortems), quarterly (relevance, target adjustments)
- Best practices: start with user-facing services, use multiple SLIs, set achievable targets (never 100%), align with business goals
- Use SLOs for prioritization: when budget is low, reliability work takes precedence over feature work

## Enhanced Knowledge: Prometheus Configuration [observability-monitoring/prometheus-configuration]
- Architecture: Applications expose /metrics -> Prometheus scrapes periodically -> feeds AlertManager, Grafana, and long-term storage
- Scrape configs: static targets, file-based service discovery (auto-refresh), Kubernetes SD with pod/service annotations
- Recording rules: pre-compute expensive queries (rate calculations, percentiles, utilization%) at defined intervals
- Key recording rules to create: request rate per service, error rate %, P95 latency, CPU/memory/disk utilization %
- Alert rules pattern: expr with threshold + `for` duration + severity label + summary/description annotations
- Essential alerts: ServiceDown (up==0), HighErrorRate (>5%), HighLatency (P95>1s), HighCPU (>80%), DiskSpaceLow (>90%)
- Naming convention: `prefix_name_unit` (e.g., `http_request_duration_seconds`); consistent labels across services
- Validation: `promtool check config prometheus.yml` and `promtool check rules *.yml` before deploying changes
- Best practices: 15-60s scrape intervals, use relabeling for metric cleanup, monitor Prometheus itself, implement HA with federation
- Long-term storage: Thanos or Cortex for retention beyond 30 days; configure retention based on storage capacity
