## Enhanced Knowledge: Hybrid Search Implementation [llm-application-dev/hybrid-search-implementation]
- Architecture: Query -> [Vector Search + Keyword Search] -> Fusion -> Results
- Fusion methods: RRF (general purpose, k=60 constant), Linear combination (alpha weighting, normalize scores first), Cross-encoder rerank (highest quality), Cascade (filter then rerank)
- RRF formula: score(doc) = sum(weight * 1/(k + rank + 1)) across result lists; no score normalization needed
- PostgreSQL hybrid: pgvector for dense search (HNSW index) + tsvector/GIN for full-text; combine with CTE and RRF in single query
- Elasticsearch hybrid: sub_searches with BM25 + kNN, use built-in RRF rank fusion (ES 8.x)
- Custom pipeline: parallel vector + keyword search, RRF or linear fusion, optional cross-encoder rerank, return top_k
- Best practices: tune weights empirically on your data, use RRF for simplicity, add reranking for significant quality improvement, log both scores for debugging, A/B test real user impact

## Enhanced Knowledge: LLM Evaluation [llm-application-dev/llm-evaluation]
- Evaluation types: automated metrics (BLEU, ROUGE, BERTScore, perplexity), human evaluation (accuracy, coherence, relevance, fluency, safety), LLM-as-judge (pointwise scoring, pairwise comparison, reference-based)
- LLM-as-judge: rate accuracy/helpfulness/clarity on 1-10 scale; pairwise comparison with winner/tie/confidence; reference-based evaluation for semantic_similarity, factual_accuracy, completeness
- Automated metrics: BLEU (n-gram overlap for translation), ROUGE (recall for summarization), BERTScore (embedding similarity), custom groundedness (NLI model for entailment), toxicity (Detoxify)
- A/B testing: use t-test for significance (alpha=0.05), Cohen's d for effect size (small<0.5, medium<0.8, large>=0.8), need sufficient sample size
- Regression detection: threshold 5% relative decrease, flag any metric that drops below baseline; integrate into CI/CD
- LangSmith integration: create datasets, define evaluators (qa, context_qa, cot_qa), run experiments with metadata tracking
- Best practices: multiple metrics for comprehensive view, representative diverse test data, always compare against baselines, statistical rigor, combine automated with human judgment, version control evaluation results

## Enhanced Knowledge: Vector Index Tuning [llm-application-dev/vector-index-tuning]
- Index type by data size: <10K -> Flat (exact), 10K-1M -> HNSW, 1M-100M -> HNSW+Quantization, >100M -> IVF+PQ or DiskANN
- HNSW parameters: M (connections per node, default 16), efConstruction (build quality, default 100), efSearch (search quality, default 50); higher values = better recall but more memory/time
- Recommendations by scale: <100K vectors (M=16, ef_construction=100), <1M (M=32, ef_construction=200), >1M (M=48, ef_construction=256)
- Quantization types: FP32 (4 bytes/dim), FP16 (2 bytes), INT8 scalar (1 byte), Product Quantization (~32-64 bytes total), Binary (dim/8 bytes)
- Memory estimation: vector_bytes = num_vectors * dimensions * bytes_per_dim; index_overhead = num_vectors * M * 2 * 4 (HNSW edges)
- Qdrant optimization profiles: "recall" (M=32, ef=256, no quantization), "speed" (M=16, ef=64, INT8 always_ram), "balanced" (M=16, ef=128, INT8), "memory" (M=8, ef=64, PQ)
- Performance monitoring: measure latency p50/p95/p99, recall@k, QPS; benchmark with real queries not synthetic; monitor recall continuously for data drift
