AUDIT SCOPE:

1. **SUPABASE RLS (Row Level Security)**

   Scan all `.sql` migration files in the project (`supabase/migrations/`, `scripts/*.sql`, any `.sql` files):
   - List every `CREATE TABLE` statement — does a corresponding `CREATE POLICY` or `ENABLE ROW LEVEL SECURITY` exist for that table?
   - Flag any table with `ENABLE ROW LEVEL SECURITY` but no policies defined (RLS enabled but no access = all rows blocked — data lockout risk)
   - Flag any table WITHOUT `ENABLE ROW LEVEL SECURITY` that stores user data (PII, auth, messages, files, billing)
   - Flag policies using `auth.uid()` inside a policy ON the `profiles` table — this causes infinite recursion. Require `SECURITY DEFINER` function approach (`get_my_role()` pattern)
   - Flag `USING (true)` or `WITH CHECK (true)` policies — these grant unrestricted access to all rows
   - Flag policies where admin check uses `EXISTS (SELECT FROM profiles WHERE ...)` inside a policy ON `profiles` — infinite recursion

2. **SQL Injection Vectors**

   Scan all Python files (`tools/*/`, `scripts/`) and JavaScript files (`tools/*/`):
   - Flag f-string SQL: `f"SELECT * FROM users WHERE id = {user_id}"` — require parameterized queries
   - Flag string concatenation SQL: `"SELECT ... WHERE " + user_input`
   - Flag `.format()` SQL: `"SELECT ... WHERE id = {}".format(id)`
   - Flag Supabase `.rpc()` calls passing unsanitized user input directly as arguments
   - Flag raw `psycopg2.execute()` or `asyncpg.execute()` calls with string interpolation

3. **Connection String Security**

   Scan all config files, `.env.example`, and source code:
   - Flag database URLs with credentials embedded: `postgresql://user:password@host/db`
   - Flag hardcoded Supabase project refs (`idorgloobxkmlnwnxbej`) in source code (not docs)
   - Verify all DB connections use environment variables: `os.environ["DATABASE_URL"]`, `process.env.SUPABASE_URL`
   - Flag Supabase service role key used client-side (in JS files that will be served to browsers)

4. **Migration File Safety**

   Scan all SQL migration files:
   - Flag `DROP TABLE` without a corresponding backup or data migration plan noted in comments
   - Flag `TRUNCATE TABLE` in a migration (destructive — should be rare and documented)
   - Flag `ALTER TABLE ... DROP COLUMN` without confirming the column is unused (grep for column name in source)
   - Flag migrations that insert or update data with hardcoded UUIDs (brittle — breaks on re-apply)
   - Flag missing `IF NOT EXISTS` on `CREATE TABLE` or `CREATE INDEX` (fails on re-run)
   - Check migrations are numbered sequentially — flag gaps in migration numbering

5. **Local Data Store Security**

   Scan `tools/*/data/*.json` and similar:
   - Flag JSON data files that contain credentials, tokens, or PII stored in plaintext
   - Flag log files (`*.log`) that echo request bodies containing auth tokens or passwords
   - Flag task queue files (`tasks/queue.json`, `tasks/registry.json`) for any entries with credential data
   - Flag SQLite databases (`.db`, `.sqlite`) — verify they're not in a web-accessible directory

6. **ORM and Query Patterns**

   Scan Python code for Supabase client usage:
   - Flag `.select("*")` without a `.limit()` on tables that could grow large — unbounded query risk
   - Flag missing error handling on Supabase client calls (`.execute()` without checking `.error`)
   - Flag `.upsert()` calls that could silently overwrite records without user awareness
   - Flag admin operations using `anon` key instead of `service_role` key (or vice versa — anon key in server-side code doing admin ops)

## Enhanced Knowledge: PostgreSQL Table Design [database-design/postgresql]
- Primary keys: prefer `BIGINT GENERATED ALWAYS AS IDENTITY`; use `UUID` only when global uniqueness/opacity needed; never use `serial`/`bigserial`
- Normalize to 3NF first; denormalize only for measured, high-ROI reads where join cost is proven problematic
- Data types: `TEXT` over `varchar(n)`/`char(n)`, `TIMESTAMPTZ` over `timestamp`, `NUMERIC(p,s)` for money (never float), `BOOLEAN NOT NULL` for booleans
- FK indexes: PostgreSQL does NOT auto-create indexes on FK columns — always add them manually; missing FK indexes cause locking issues on parent deletes
- JSONB: prefer over JSON; index with GIN; use `jsonb_path_ops` opclass for containment-only workloads; extract scalar fields to generated columns for B-tree indexing
- UNIQUE + NULLs: standard allows multiple NULLs; use `NULLS NOT DISTINCT` (PG15+) when only one NULL should be allowed
- Enums: `CREATE TYPE ... AS ENUM` for small stable sets; use TEXT + CHECK or lookup table for evolving business values
- Partitioning: use for >100M rows; RANGE for time-series, LIST for discrete values, HASH for even distribution; no global UNIQUE without partition key
- Safe schema evolution: transactional DDL, `CREATE INDEX CONCURRENTLY` avoids write locks, volatile defaults cause table rewrites
- Row-Level Security: `ALTER TABLE ENABLE ROW LEVEL SECURITY` + `CREATE POLICY`; avoid `auth.uid()` queries against the `profiles` table (infinite recursion)

## Enhanced Knowledge: SQL Optimization Patterns [developer-essentials/sql-optimization-patterns]
- Always use `EXPLAIN (ANALYZE, BUFFERS)` to understand query plans; key metrics: Seq Scan (usually slow for large tables), Index Scan (good), Index Only Scan (best)
- Index types: B-tree (default, equality/range), GIN (JSONB/arrays/full-text), GiST (ranges/geometry), BRIN (large naturally ordered data, minimal storage)
- Composite index column order matters: leftmost prefix must match query conditions; put most selective columns first
- Covering indexes with INCLUDE: `CREATE INDEX ON tbl (id) INCLUDE (name, email)` enables index-only scans
- Partial indexes for hot subsets: `CREATE INDEX ON tbl (user_id) WHERE status = 'active'`
- N+1 query elimination: use JOINs or batch loading with IN clause instead of loops; never call `.select()` inside a loop
- Pagination: avoid OFFSET on large tables; use cursor-based pagination with `WHERE (created_at, id) < (last_cursor_values)` + matching index
- Aggregation: use `pg_class.reltuples` for approximate counts; filter with WHERE before GROUP BY; use covering indexes for aggregations
- Correlated subquery to JOIN transformation: replace `(SELECT COUNT(*) FROM orders WHERE user_id = u.id)` with `LEFT JOIN + GROUP BY`
- Monitoring: `pg_stat_statements` for slow queries, `pg_stat_user_tables` for missing indexes (high seq_scan), `pg_stat_user_indexes` for unused indexes

## Enhanced Knowledge: Database Migration [framework-migration/database-migration]
- Every migration needs both `up()` and `down()` (rollback); test rollback procedures on staging before production
- Zero-downtime column rename: (1) add new column, (2) dual-write, (3) backfill, (4) switch reads, (5) drop old column
- Column type changes on large tables: add new column, copy+transform data, drop old column, rename new — never ALTER COLUMN TYPE directly on large tables
- Transaction-based migrations: wrap in `BEGIN/COMMIT` with rollback on error; use checkpoint-based rollback with backup tables for large data migrations
- Break migrations into small, incremental steps; make each step backward-compatible with running application code
- Always backup before migration; use `IF NOT EXISTS` on CREATE statements for idempotency
- Handle NULL values explicitly during data transformations; consider index performance impact of schema changes
- For large data backfills, process in batches to avoid long-running transactions and lock contention
- Cross-database considerations: handle JSONB vs JSON, type differences, and dialect-specific SQL in migration scripts
- Common pitfalls: not testing rollback, breaking changes without downtime strategy, forgetting NULL handling, ignoring FK constraints during migration
